# Project Context for AI Assistants

## Overview

This repository manages a full homelab Kubernetes infrastructure on **Talos Linux v1.12.2 / Kubernetes v1.35.0**, running on a TrueNAS SCALE host via QEMU/KVM VMs. All infrastructure is managed with **Terraform** and **Helm**. No manual kubectl applies except for bootstrapping. This is what is called production environment

## Cluster

- **Talos Linux** v1.12.2, **Kubernetes** v1.35.0
- 3 control plane nodes, 2 worker nodes (6 GB RAM each)
- **CNI**: Flannel
- **Load balancer**: MetalLB (L2 mode)
- **Ingress**: Istio 1.29.2 with Gateway API (standard + experimental channel)

There are two Terraform root modules:

| Module            | Purpose                                                                |
| ----------------- | ---------------------------------------------------------------------- |
| `terraform-base/` | Core infrastructure: storage, networking, secrets, certificates        |
| `terraform-apps/` | Applications: Immich, Plex, Komga, Prometheus, SeaweedFS, Velero, CNPG |


### Terraform Backend
- Two backends: `backend.prod` (S3 via SeaweedFS) and `backend.local` (local file)
- Switch with `../tfinit.sh prod` or `../tfinit.sh local`

## Running Terraform

```bash
# Initialise (prod = S3 backend, local = local file)
cd terraform-base && ../tfinit.sh prod

# Apply with 1Password secret injection
cd terraform-base && op run --env-file=envs/.env.prd --no-masking -- terraform apply

# Generate module docs
./tfdocs.sh
```

## Conventions

### Modules

- Every module follows standard Terraform module layout: `main.tf`, `variables.tf`, `outputs.tf`, `README.md`
- Terraform Modules always have variables with descriptions. It has examples with instructions on how to use inside the description attribute if variable has no default value. If description is multine line, use heredoc sintax with <<--EOT description EOT
- Variables with no default must include usage examples in their description
- Terraform documentation is generated by using ./tfdocs.sh from the kubernetes folder

### Helm Releases OR kubernetes deployments
- Always use `helm releases` to install existing charts. If chart doesn't exist use `kubernetes` provider resources like `kubernetes_deployment_v1` and etc. Lastly if no `kubernetes` resources exists use `kubernetes_manifest`
- Every `helm_release` has a `chart_version` variable
- Helm values always use `<<-EOF ... EOF` heredoc syntax — never `yamlencode()`
- Every release sets `part-of` and `component` labels
- Every release configures pod affinity based on the `component` label
- Resource requests/limits are always exposed as flat, independent variables (never nested objects)
- Resource variable defaults should be the minimum viable values

### Labels
```hcl
labels = {
  part-of   = "subsystem-name"   # e.g. "monitoring", "secrets", "backup"
  component = "specific-role"    # e.g. "prometheus", "grafana"
}
```

### Secrets Management
- If `vault_password != null`: use **Vault CSI Secret Provider Class** to inject secrets
- If `vault_password == null`: generate a random secret with `ephemeral "random_password"` and store via `kubernetes_secret_v1` using `data_wo` / `data_wo_revision` (keeps secrets out of state)
- Never store plaintext secrets in Helm values or Terraform state
- Local secret injection uses `op run --env-file=envs/.env.prd` with `op://` URI references

## Secret Injection Pattern (Vault CSI)

Every module that uses Vault follows this pattern:

1. `vault_policy` — grants read on the secret path
2. `vault_kubernetes_auth_backend_role` — binds the policy to a service account + namespace
3. `SecretProviderClass` (via `extraManifests` or `extraObjects`) — maps Vault fields to Kubernetes Secret keys
4. Pod mounts the CSI volume at `/mnt/secrets-store`


### Storage
- NFS CSI driver (`nfs.csi.k8s.io`) for all persistent volumes
- Two storage classes:
  - `persistent` — `Retain` reclaim policy, for stateful workloads
  - `default` — `Delete` reclaim policy, marked as default
- Static NFS PVs use `volume_handle` and `volume_attributes.share`
- Always set `securityContext` when creating PVs
- When creating a persistent volume always use minimal data disk

### Gateway / Routing
- All modules receive a `gateway` variable: `{ name, namespace }`
- Every module creates an `HTTPRoute` (or `TCPRoute`) pointing to its gateway

### CRDs and Custom Resources
- Prefer to use native helm directives if present in the helm chart instead of creating resources. If it's not possible use `extraObjects` or `extraManifests` in Helm values over `kubernetes_manifest` for CRDs
- Use `kubernetes_manifest` only when no Helm alternative exists


## Key Components

### Networking
- **MetalLB** (L2): assigns IP to Istio gateway
- **Istio** + **Gateway API**: all HTTP/TCP routing; `HTTPRoute` and `TCPRoute` resources
- **cert-manager**: three issuers — uploaded CA, Vault PKI, Let's Encrypt (Cloudflare DNS-01)

### Secrets
- **HashiCorp Vault** (HA, 3-node Raft): primary secret store
- **1Password Connect**: Vault backend via custom plugin
- **Secrets Store CSI Driver** + Vault provider: syncs Vault secrets into Kubernetes `Secret` objects
- Vault Kubernetes auth backend is used by all workloads; each module creates its own `vault_policy` and `vault_kubernetes_auth_backend_role`

### Storage
- **SeaweedFS**: S3-compatible object store for Velero backups and Terraform state
- **NFS CSI driver**: dynamic and static NFS-backed PVs from TrueNAS
- **CloudNativePG (CNPG)**: PostgreSQL operator with backup to SeaweedFS

### Observability
- **kube-prometheus-stack**: Prometheus, Grafana, Alertmanager, kube-state-metrics, node-exporter
- Grafana credentials via Vault CSI or auto-generated random password

### Applications (`modules/home-apps/`)
- **Immich** (`immich-photos` namespace): photo management via bjw-s Helm chart
- **Plex**: media server with NFS-backed volumes for movies, music, TV shows
- **Komga**: ebook/comic reader
